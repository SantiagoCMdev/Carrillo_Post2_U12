# ADR-002: Factory + Strategy para selección de procesador de pedidos

## Estado
Aceptado — 2026-05-07

## Contexto
Existen tres tipos de pedido (Estándar, Express, Internacional) con algoritmos de cálculo de costo distintos. Un bloque `if/else if` en el servicio principal violaba el principio Open/Closed y requería modificar el servicio cada vez que se agregara un nuevo tipo de pedido.

## Decisión
Se usa el patrón Strategy (interfaz `ProcesadorPedido` con una implementación por tipo de pedido) combinado con Factory (`ProcesadorPedidoFactory` que recibe todas las implementaciones inyectadas por Spring en una lista y las indexa en un `Map<TipoPedido, ProcesadorPedido>`) para selección dinámica en tiempo de ejecución.

## Consecuencias

**Positivas:**
- Agregar un nuevo tipo de pedido requiere solo crear una nueva clase `@Component` sin modificar código existente.
- Cada procesador es testeable de forma independiente con pruebas unitarias simples.
- La CC del método principal bajó de 4 a 1.

**Negativas:**
- Más clases que un switch centralizado, lo que puede parecer sobre-ingeniería para tres casos.
- La selección implícita por mapa puede sorprender a desarrolladores nuevos que no conocen el patrón.
